分段

上个章节讲述了实模式寻址，在本章我们将介绍保护模式提供的第一中机制，段模式寻址。在现代的处理器中，内存管理通常提供了分段和分页机制。分段可以将程序代码，数据，栈按照模块划分，并且运行在相同的物理内存空间。每个任务都可以有自己的代码，数据和栈段等等。分页主要提供了虚拟内存管理，按需所取得内存管理模式，同时分页也用于多任务的隔离。

在保护模式下，分页被默认使用，没有任何方式可以禁用分段。但是，处理器提供了机制可以关闭分页。也就是说分页是可选的。而分段是强制的。这两种机制可以共享内存的单任务，多任务和多处理器系统。正如下图描述

分段提供了一种将处理器可寻址空间划分为较小的受保护的地址空间（段）的机制。段可以用于保存代码，数据和栈。也可以用于保存系统数据结构（TSS，LDT等）。在多任务的系统中，可以为每一个任务分配一组段。处理器来保证写入一个任务的段不会影响另外一个任务的的执行。而且处理器还可以限制对特定类型分段执行操作，从而提供保护功能。

为了定位内存中一个一个字节，应用程序必须听一个逻辑地址（一个远指针）。逻辑地址由一个段选择器和一个偏移量组成。通过使用段选择子作为段描述符表的偏移量，定位到段描述符（每个段描述符一般包含段的大小，访问权限，段类型，特权级别等等）。从段描述符表得到段的基址加上偏移量从而获得线性地址。

如果分页机制没有打开，那么这个线性地址地址将会直接映射为物理地址。

在多任务系统中，每一个任务通常都会有一个地址空间，所有任务的地址空间通常远远大于可用的内存空间。为了解决这个问题，处理器虚拟化了地址空间。而这种虚拟化主要通过分页来实现。分页机制可以使用少量的内存空间在模拟出程序所需要的大内存空间，每一个任务独享自己的虚拟内存空间。通常操作系统负责维护分页机制，比如，操作系统为应用程序分配页表和页目录，应用程序访问一个虚拟内存地址，处理器会根据逻辑地址，页目录，页表共同映射转化为最终的物理内存，然后在对应的物理位置执行读或写。操作系统通常都支持按需所取，也就说如果操作系统发现对应的物理页不再内存中，处理器将会触发一个缺页异常（page-fault exception）。最终由操作系统负责将对应的页重新调入内存。应用程序最终获取对应的数据而感知不到这些过程。

32位处理器支持的分段机制可以用于各种系统设计。这些设计从平坦模型（仅使用最小限度地使用分段来保护程序）到多分段的模型，

Intel的分段机制提供了更加强大的分段模型，参考Intel开发手册，我们这里只讲述分段的平坦模型：

  • 平坦模型 - 使用分段机制提供的最简单的内存模型，这种模型将不再采用分段机制，变相的绕过了分段机制，将内存划分位一个连续的字节数组。它为系统开发人员和应用开发人员隐藏了系统的分段机制。这个模型也是Linux所采用的模式。在这个模式中，系统创建了连个段描述符，一个用于代码段，一个用于数据段。但这连个地址都映射到整个线性地址空间。也就说，在32位系统中将这两个段有相同基址0和相同的段限制4GB。这两个段共享了同一个内存地址空间。

64位中的分段

在64位的处理器中，分段根据运行的模式不同而有所区别，在兼用模式中，分段机制和32位中一致。在64位中，段管理通常默认关闭，处理器默认创建一个64位的平坦线性地址空间。处理器默认将CS, DS, ES, SS设置为0，从而创建一个和有效地址相同的线性地址，而FS和GS段寄存器是例外，这两个段寄存器可以用于特殊目的。比如用于寻址某些操作系统的关键数据结构。

另外，在64位中，处理器不会再运行的时候检查段限制。

物理地址空间

处理器通过地址总线进行寻址，地址总线的宽度决定了可以寻址的最大空间。保护模式中，32位处理器可以提供最大4GB（2^32 Byte）的物理地址空间。我们可以将物理地址想想为连续的字节数组，索引作为其内存地址。物理地址可以映射成读写存储器，只读存储器和I/O地址。64位处理器通过CPUID指令可以获取处理器支持的最大物理内存宽度。

逻辑地址和线性地址

保护模式下，处理器通过使用逻辑地址转译和线性地址分页两个阶段获取最终的物理地址。逻辑地址和线性地址都是虚拟地址。

逻辑地址是由逻辑地址转译为线性地址的重要参与者，我们引用的任何一个地址都可以看做为逻辑地址（通常也叫做虚拟地址），一个逻辑地址由16位段选择子和32位偏移量组成，而在汇编语言我们只看到的地址是偏移量，一般不同类型的数据会默认加载不同的段寄存器，数据加载的是数据段寄存器，代码段加载的是代码段寄存器。段选择子用于选择段描述符表中的的段描述符，从段描述附中获得基址和偏移量最终组成线性地址。

而线性地址由逻辑地址转译而来，它也是一种虚拟地址，32位处理器最大寻址从0到2^32-1。在平坦内存模型中，系统中所有的数据都存放在这个地址空间中。如果没有打开分页，那么线性地址就是物理内存中的物理地址。常见的线性地址是一个32位的地址。如果分页打开，还需要经过一层分页才能得到最终的物理地址，这个也是现代操作系统中能够实现虚拟内存概念的重要支撑。

段选择子

段选择子是一个为了定位段描述符的16位的标识符。段选择子不会直接指向段描述符，而是GDT（全局描述符表）或LDT（本地描述符表）的偏移量，通过这个偏移量从而获得段描述符参与地址寻址。正如在图中展示，段选择子包含以下部分

  • Index -（3到15位），用于选择GDT（全局描述符表）或LDT（本地描述符表）中的段描述符，处理器 Index * 8 + GDT/LDT来获得段描述符的地址。

  • TI 标志 - （第2位），0表示GDT（全局描述符表），1表示LDT（本地描述符表）

  • Requested Privilege Level（RPL） - （0到1位），这是一个很重要的概念。它指定了选择子的特权级别。特权级别从0到3，其中0是最高级。

需要注意的是，GDT的第一个段选择符没有被使用，而是作为保留。如果段选择子的Index为0，TI标志也是0，那么这个段选择子叫做空段选择子。除了CS和SS寄存器加载空段选择符会抛出#GP异常，其他的段寄存器并不会抛出异常，但用空段选择子访问内存会导致抛出异常，并且空段选择子可以用于初始化未使用的段寄存器。在64位处理器上，处理器不会对空段选择子执行运行时检查，尝试访问有空段选择子的内存访问并不会引起#GP异常。

段寄存器

段寄存器是历史的产物，根据约定，通常将 CS 段寄存器用于代码段，将 DS 用于数据段，将SS 用于栈段。处理器也提供了ES，FS，GS等段寄存器用于其他目的。在早期的处理器中，段寄存器作为分段直接使用，在现代的寄存器中，段寄存器作为段选择子使用，并且段寄存器分为可见部分和不可见部分，不可见部分主要用于地址转换的加速功能。不可见部分是没有办法修改的，有处理器隐含的进行加载，它可以作为段描述符的缓存部分。

在64位处理器中，ES，DS，SS 段寄存器不再使用，这些集群器描述符中的基址，段大小等都会被忽略，引用这些段寄存器默认都会将其基址作为0处理。而且处理器也不会执行段范围的检查，替代的是执行 规范地址 检查。

在64位模式下例外的是 FS，GS段寄存器，它们还是采用 (FS / GS).base + index + displacement 的寻址模式，使用这种方式处理器也不会执行运行时检查。处理器将 (FS / GS) 寄存器隐藏部分映射到MSR中，可以使用 WRMSR 指令修改 (FS / GS).base。在新的处理器中，处理器提供了 SWAPGS 指令用于替换 GS段寄存器的base部分。这两个段寄存器加载的地址必须是规范化的，否则处理器将抛出 #GP 错误。

段描述符

段描述符是处理器提供段的基址和段大小以及控制信息等数据结构，它通常位于 GDT/LDT 表中。段描述符我们重点关心的是基址，段大小，DPL域信息，有关段描述符详细的信息参考Intel开发手册。

逻辑地址到线性地址

处理器为了访问内存中的一个字节，需要经历逻辑地址到线性地址的转换：

  1. 使用段选择子从GDT（全局描述符表）或LDT（本地描述符表）中读取段描述符

  2. 处理器检查段描述符的访问权限，确保访问地址在段描述符中的有效范围，确保没有非法访问等

  3. 将从段描述符中获取的基址加上内存访问的偏移量获得最终的线性地址

以上描述的是基于32位地址的转译，而在64位中也和上面的步骤差不多，区别是偏移量变为了64位，可寻址的地址空间也增加为了64位。

处理器访问内存中的任何一个字节都会经历将逻辑地址转换为线性地址。在下面的这个例子就是一段代码段的寄存器，默认的段寄存器对应的是CS，偏移量为ffffffff81000040，由于linux中采用的是平坦内存模型，CS寄存器默认的基址为0，所以最终转译的线性地址为ffffffff81000040，这个地址参与最终的分页转换。

ffffffff81000040 <secondary_startup_64>:
ffffffff81000040:       e8 cb 00 00 00          callq  ffffffff81000110 <verify_cpu>
